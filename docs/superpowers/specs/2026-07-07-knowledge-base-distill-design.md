# 知识库提炼（发布脱敏）设计

> 把 `知识库/` 下 63 篇已整理文档做二次提炼，剔除人物署名/头衔/采访人、删整理者旁白小节、软化互链，产出适合公开发布的知识内容。**原地覆盖**，分批推进，git 可回滚。

## 背景

`知识库/`（6 分类 63 篇结构化 markdown）是 server `KnowledgeService` 的运行时数据源 —— 启动时扫 `<分类>/*.md`，解析 frontmatter `tags` + 开头「一句话」作 description，内存缓存，经 `/knowledge` 给 agent-ui 浏览页与 curator agent 的 `list_knowledge`/`get_knowledge` 工具使用。

这批文档源自 OCR/抽取的访谈、讲座、教程，整理时保留了三类"内部痕迹"，不适合作为公开发布内容：

1. **人物署名与头衔引用** —— `**血红**（坚持论）：「…」`、表格 `| **血红** | 习惯 |`、"起点主编"/"失落叶（网游）"/"老宫编辑"等。发布版需脱敏成无主语客观陈述。
2. **末尾 `### 🧠 整理思考` 小节** —— 整理者的结构判断/知识提炼/使用建议/关联（带 `[[文件名]]` 互链）。属"关于文档的元讨论"，发布版应删，有用的增量融入正文。
3. **行内 `> 🧠 …` 旁注 + `[[文件名]]` 互链** —— 机器/整理者标记，发布版需软化成自然文本。

## 目标

- 63 篇全部提炼，**原地覆盖** `知识库/<分类>/*.md`
- 剔除人物署名/头衔/采访人，改写为无主语客观陈述
- 删末尾 `### 🧠 整理思考` 整节（有用增量融入正文）
- 行内 `> 🧠` 去标记融为正文；`[[文件名]]` 软化为 `《文件名》` 文本
- 保留 frontmatter `tags`、开头 TL;DR、标题/表格/金句、作品名/平台名/数据案例

## 非目标

- 不新建发布版目录（用户选定原地覆盖）
- 不删作品名（《鬼吹灯》等）/平台名（起点）/具体数据（月票数）—— 这些是案例锚点，保留
- 不改 server `KnowledgeService` 代码（保持扫 `<分类>/*.md` 不变）
- 不引入语义检索/向量化
- 拆解小说模块（`/dissect`）、对标库（BenchmarkBook）的发布脱敏不在本期

## 提炼规格

### 删 · 人物署名与头衔（逐篇语义判断）

| 现状模式 | 改写 |
|---|---|
| `**血红**（坚持论）：「现在网络创作环境不好…」` | `坚持：网络创作环境虽不好，但既然投入就应坚持。扛住攻击批评的耐力是动笔前提；初期不以物质利益为目标，当爱好坚持。` |
| 表格 `\| **血红** \| 不写大纲、不构思… \|` | 去"谁"留"知识"：`灵感流写法（天才型特例，新人不宜模仿）：不写大纲、随手流出文字、码字 6500–7500 字/时…` |
| "各路大神习惯对照表"（本质是"谁的习惯"） | 整表重构成"常见创作习惯"知识列表 |
| `起点主编` / `失落叶（网游）` / `老宫编辑` | 头衔作来源标记→删头衔留陈述；"XX 说"直接引语→转客观陈述 |

**判断准则（批 1 对齐后细化）：**
- **作品名 / 平台名 / 数据** → 留（案例锚点）
- **人物名 / 笔名 / 头衔 / 采访人** → 删，知识改无主语陈述
- **边界**：以人名命名的分类组织（如"血红创作观"）→ 拆成知识点，去人名
- **开头 TL;DR blockquote 与 frontmatter** 也按此规则审：TL;DR 文本里的人名一并脱敏；`tags` 原则保留，但若某 tag 本身是人名/头衔则替换为中性分类

### 删 · 末尾 `### 🧠 整理思考` 整节

- 结构判断 / 关联 / OCR 过程等**元讨论** → 弃
- 知识提炼 / 使用建议若有**正文未覆盖的增量** → 融入对应段落（不留🧠标记、不单列小节）
- 判断主观时默认弃，仅明确未被正文覆盖的知识点才融入

### 改 · 行内 `> 🧠 …` 旁注

去 `> 🧠` 标记（含前缀 `> 🧠 ` 与单独成行的 `> 🧠`），内容融为正文普通段落/列表项。

### 改 · `[[文件名]]` 互链

`[[网文入门与签约指南]]` → `《网文入门与签约指南》`（纯文本书名号，非链接）。

### 留 · 知识与结构

- frontmatter `tags`：**必须保留**（原地覆盖，`KnowledgeService` 靠它检索）
- 开头 blockquote「一句话 + 写作价值/适用场景」TL;DR：保留（人名脱敏）
- 分级标题、表格、金句范例：保留
- 作品名（《鬼吹灯》《小兵传奇》）、平台名（起点）、具体数据（月票 107 票）：保留
- 已订正的 OCR 错字：保持

## 执行结构（机械 / 语义 / 校验分离）

63 篇里大量工作是重复的机械清理，脚本确定性完成；只有"署名→陈述"需逐篇判断。

```
1. 备份   git add 知识库/ && git commit                     ← 可回滚底线
2. 机械   脚本 scripts/distill_kb/clean_mechanics.mjs:
            · 删末尾「### 🧠 整理思考」整节（该标题到下一篇节标题/文件尾）
            · 行内「> 🧠 X」→「X」（去 "> 🧠 " 前缀留内容）
            · 「[[文件名]]」→「《文件名》」
          确定性，全量一次跑完 + commit
3. 语义   逐篇 LLM 重写（分批）:
            · 署名/头衔 → 无主语陈述
            · "谁的习惯"表格 → 知识列表重构
            · 判断作品名是案例(留) vs 署名(去)
            · 整理思考小节有用增量 → 融入正文
            · 开头 TL;DR + frontmatter tags 人名脱敏
4. 校验   脚本 scripts/distill_kb/lint.mjs:
            · 无残留 🧠 / [[ ]]
            · frontmatter tags 还在（每篇有非空 tags）
            · 非空 / 字数降幅 ≤ 40%（防误删过多）
            不通过 → 标记该篇复查
```

**为何不选纯手工逐篇**：机械部分（删小节/转互链/去🧠）每篇重复，63 篇手工易漏、慢、无校验。脚本做确定性部分、LLM 做判断部分、lint 兜底一致性。脚本是一次性工具（node，无依赖），跑完即可，不进任何 build/CI。

| **批 1（样板）** | 方法论教程 1 + 拆文案例 3 + 创作须知·上架审核 1 | 5 | 访谈/运营型，署名最密集；**先做完对齐判断准则** |

每批流程：机械清理（已全量跑过则跳） → 逐篇语义重写 → lint → **git commit**（分批提交，粒度回滚）。

## 风险与缓解

| 风险 | 缓解 |
|---|---|
| 改的是 agent 数据源，破坏 KnowledgeService | git 分批提交，每批独立回滚；lint 保证 tags/结构不破 |
| "案例 vs 署名"误判（如作品名误删） | 批 1 做完先对齐准则再推批 2–5 |
| 风格漂移 / 误删过多 | lint 字数降幅阈值 + 批尾抽检 |
| 整理思考小节"有用增量"判断主观 | 默认弃，仅明确未被正文覆盖的知识点才融入 |
| 机械脚本误伤（如把正文里的 `[[ ]]` 当互链） | 脚本只匹配 `\[\[[^\]]+\]\]`；跑完 lint + 抽检 |

## 验收标准

- 63 篇全部通过 lint：无 🧠 / 无 `[[]]` / tags 在 / 非空 / 字数降幅 ≤ 40%
- 批 1 抽检：署名已删、作品名保留、陈述通顺、整理思考小节已消失、TL;DR 已脱敏
- `KnowledgeService` 仍能正常扫描（tags + description 解析不变）—— server 启动或既有知识库相关单测验证
- git 历史清晰：每批一个 commit，可粒度回滚

## Out of scope

- 不新建发布版目录、不改 KnowledgeService、不动 `整理规范.md`/`kb_index.md`/`_stats.md` 的 meta 性质（其内若含人名，仅做最小脱敏）
- 不删作品名/平台名/数据
- 不做语义检索 / 向量化
- 拆解小说模块、对标库的发布脱敏不在本期
