# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository Layout

`narratox` is a multi-project repo. The **root `package.json` only orchestrates** the sub-projects — it runs them in parallel via `npm-run-all2` (`run-p dev:*` / `run-p build:*`). It is **not a pnpm workspace** and shares no dependencies, so each sub-project still needs its own `pnpm install` (and the root needs `pnpm install` once, for `npm-run-all2`).

- **`agent-ui/`** — A Next.js 15 (App Router) + React 18 + TypeScript **AI novel-writing workspace** (the dedicated client for `server`). Novel library, a 3-zone writing workspace (IconRail / chat / resource panel), 拆解小说 (dissection) module, 写作知识库 browser, login/register, and a settings page.
- **`server/`** — A NestJS 11 + TypeScript + Prisma 7 (PostgreSQL) API. JWT auth, two agent runtimes (the `deep-agent` writing agent + a separate `dissect` agent), per-user/per-agent configurable multi-provider models (Vendor/Model two-tier), novel/chapter CRUD, novel publishing, and a global 写作知识库.
- **`design/`** — **The Pencil design source of truth for the whole UI.** Holds `narratox.pen` (the encrypted Pencil design file — 38 top-level screen frames covering auth / library / knowledge / dissect / settings / workspace + 2 reusable components `AppSidebar` / `IconRail`) and `_exports/` (HTML/PNG reference exports). **This directory is modified ONLY via the Pencil MCP tools** (`get_editor_state` / `batch_get` / `batch_design` / `snapshot_layout` / `get_screenshot` / `get_variables` / `export_nodes` / `export_html`). `.pen` files are encrypted — **never `Read`/`Grep`/`cat` them directly**; always go through Pencil MCP. See [Architecture › design (Pencil MCP)](#design-pencil-mcp) below. Design tokens authority: [docs/superpowers/specs/2026-07-02-ui-redesign-design.md](docs/superpowers/specs/2026-07-02-ui-redesign-design.md).
- **`知识库/`** — The global writing knowledge base (6 categories: 人设档案 / 公式模板 / 创作须知 / 拆文案例 / 词汇素材库 / 方法论教程). Read-only markdown served to agents + the `/knowledge` FE by `KnowledgeService` (path overridable via `server`'s `KB_DIR` env, default `<repo>/知识库`). Not built, gitignored content is the corpus.
Package manager is **pnpm** everywhere.

## The two app projects ARE connected

`agent-ui` is the locked front-end for `server` (no external AgentOS anymore):

- **`agent-ui` talks to `server`.** Its endpoint defaults to `http://localhost:3001` (set as `selectedEndpoint` in [agent-ui/src/store.ts](agent-ui/src/store.ts)). Auth is JWT — the token lives in the same Zustand store and is sent as `Authorization: Bearer <token>`.
- **Ports:** `agent-ui` dev runs on `:3000` (`next dev -p 3000`). `server` defaults to `process.env.PORT ?? 3000`, but the root `dev:server` script pins `PORT=3001`, so `pnpm dev` at the root runs both without collision — agent-ui → `:3000`, server → `:3001`. Running `pnpm --dir server start:dev` directly still uses `:3000`.
- `server/.env` (gitignored) must define `DATABASE_URL`, `JWT_SECRET`. **Model provider API keys are NOT in `.env`** — they are configured per-user in the `/settings` UI under a `Vendor` (provider/baseUrl/apiKey) with one or more `Model` rows; the key never leaves the server). See `server/.env.example` (which also documents the optional `KB_DIR` override for the global 知识库).

## Common Commands

### root (repo root — orchestration only)
```sh
pnpm install            # once: installs npm-run-all2 at the root
pnpm dev                # run-p dev:* → agent-ui (:3000) + server (:3001) in parallel
pnpm build              # run-p build:* → both builds in parallel
pnpm dev:agent-ui       # just agent-ui
pnpm dev:server         # just the server
```

### agent-ui (`cd agent-ui`)
```sh
pnpm dev                # next dev -p 3000  → http://localhost:3000
pnpm build              # next build
pnpm start              # next start (production)
pnpm lint               # next lint
pnpm lint:fix           # next lint --fix
pnpm typecheck          # tsc --noEmit
pnpm format             # prettier --check (does NOT write)
pnpm format:fix         # prettier --write
pnpm validate           # lint && format && typecheck  (CI-style gate)
```
There is **no test runner configured** in `agent-ui` (no Jest/Vitest/Playwright). The quality gate is `pnpm validate`.

### server (`cd server`)
```sh
pnpm start:dev          # nest start --watch (root dev:server pins PORT=3001)
pnpm build              # nest build -> dist/
pnpm start:prod         # node dist/main
pnpm test               # jest unit tests (NODE_OPTIONS=--experimental-vm-modules — needed for ESM deepagents mocks)
pnpm typecheck          # tsc --noEmit
pnpm test:cov           # coverage
pnpm lint               # eslint --fix
pnpm format             # prettier --write

# single test
pnpm test -- novel.service.spec.ts      # by file
pnpm test -- -t "should return ..."     # by test name
```
There is **no e2e runner** — only the jest unit suite (`src/**/*.spec.ts`). Specs mock collaborators via constructor injection with plain `jest.fn()` doubles (see `novel.service.spec.ts`, `model-config.service.spec.ts`); model-factory routing is tested as a pure function (`model-factory.spec.ts`). **Prisma 7 is config-driven** (`server/prisma.config.ts`), so prisma CLI commands take no `--schema` flag.

## Architecture

### agent-ui (Next.js — novel workspace)
App Router. Path alias `@/*` → `agent-ui/src/*` (in `tsconfig.json` and `next.config.ts`).

> **Layout note (updated 2026-07-02):** the workspace FE is `[IconRail ~56px] [ChatPanel flex-1] [ResourcePanel ~420px]`; the library (`/`) / knowledge (`/knowledge`) / settings (`/settings`) pages share a `components/layout/AppSidebar.tsx`; the workspace `IconRail` no longer has a settings entry. The earlier swarm-era `ResourceNav` / `ChapterPreview` / `WritingChapter` bullet specifics below are superseded — the **design source of truth is now `design/narratox.pen`** (see [Architecture › design (Pencil MCP)](#design-pencil-mcp)). The `ChapterPreview` → right pane is now `ResourcePanel` with ~10 switchable resource views (info/worldview/references/outline/chapters/characters/hooks/events/overview/voiceProfile).

- **Routes** — [src/app/](agent-ui/src/app/): `/` novel library ([page.tsx](agent-ui/src/app/page.tsx) → `components/library/`), `/login` + `/register` (`app/(auth)/`), `/novels/[id]` workspace ([page.tsx](agent-ui/src/app/novels/[id]/page.tsx) → `components/workspace/`), `/dissect` ([page.tsx](agent-ui/src/app/dissect/page.tsx) → `components/dissect/DissectPage.tsx` — 对标拆解 module: list / upload / log drawer / result browser), `/knowledge` ([page.tsx](agent-ui/src/app/knowledge/page.tsx) → `components/knowledge/KnowledgeBrowser.tsx` — global 写作知识库 browser), `/settings` ([page.tsx](agent-ui/src/app/settings/page.tsx)). Every protected page is wrapped in [RequireAuth](agent-ui/src/components/auth/RequireAuth.tsx) (probes `/auth/me`; 401 → logout + redirect). 「新建小说」in the library calls `POST /novels` (bare CONCEPT novel, title 占位) and navigates to `/novels/[id]`.
- **Workspace (3-zone, the core)** — `components/workspace/`:
  - `ResourceNav` (left) — back-to-library, novel title, **小说信息卡** (title / genre / worldviewText / style, refreshed every turn-end so onboarding fills it in live), and greyed **P2/P3 placeholders** (大纲/角色/世界观/状态). The chapter list + 「新章」button are **gone** (v0.4.0) — chapters are AI-driven via `write_chapter`. Adding a future resource = a new nav section + a new right-pane view.
  - `ChatPanel` (center, invariant) — **reuses the existing chat infra** (`MessageArea` + `ChatInput` + `useAIStreamHandler`). On mount it sets nuqs `agent=<workspace-agent>` / `session=<novel.sessionId>` / `db_id=default` so the streaming hook posts runs to `POST /agents/:id/runs` (mode `workspace`) against the novel's chat thread. **No「采纳」button** — the writer Agent auto-writes chapters server-side via the `append_section` tool (et al.); the manuscript pane refreshes on turn-end.
  - `ChapterPreview` (right, renamed from `ChapterDetail` in v0.4.0) — preview pane for AI-written chapters: a `‹ 第 N 章 ›` switcher (browse any chapter read-only), a **skeleton-loading state** triggered by the `WritingChapter { order:N }` signal (auto-jumps to that chapter while the writer is working), and the rendered manuscript (MarkdownRenderer) once the stream ends. Plain-text edit mode still saves via `PATCH /novels/:id/chapters/:cid`. While the novel is still `CONCEPT` (no chapter content yet) it shows a「立项中」placeholder.
- **API layer** — [src/api/routes.ts](agent-ui/src/api/routes.ts) maps operations to server URLs (`/novels` + sub-resources, `/agents/:id/runs`, `/sessions/...`, `/auth/...`, `/settings/vendors.../models...` + `/settings/agent-models` + `/settings/voice-profiles`, `/knowledge`, `/benchmarks` + `/benchmarks/:id/{dissect,stream,upload}`, `/health`). Clients: [src/api/novels.ts](agent-ui/src/api/novels.ts) (novel/chapter), [src/api/auth.ts](agent-ui/src/api/auth.ts), [src/api/os.ts](agent-ui/src/api/os.ts) (sessions + status), [src/api/settings.ts](agent-ui/src/api/settings.ts) (vendors/models/agent-models/voice-profiles), [src/api/knowledge.ts](agent-ui/src/api/knowledge.ts), [src/api/benchmark.ts](agent-ui/src/api/benchmark.ts). Novel/chapter types in [src/types/novel.ts](agent-ui/src/types/novel.ts) (note: `NovelListItem` vs `Novel` — only the detail/get response includes `chapters`).
- **Streaming** — Agent runs stream newline-delimited JSON. [src/hooks/useAIResponseStream.tsx](agent-ui/src/hooks/useAIResponseStream.tsx) is the incremental JSON parser; [src/hooks/useAIStreamHandler.tsx](agent-ui/src/hooks/useAIStreamHandler.tsx) turns frames into `store.messages`. **RunContent frames carry CUMULATIVE content** (each frame = full text so far); the handler extracts deltas via `chunk.content.replace(lastContent, '')` and sets the clean final text on `RunCompleted`. **`WritingChapter { order:N }`** is a side-channel signal frame (v0.4.0) emitted when the writer Agent calls `write_chapter` — the handler uses it to auto-jump `ChapterPreview` to chapter N and show the skeleton while the write streams.
- **State** — One Zustand store in [src/store.ts](agent-ui/src/store.ts) (`useStore`), persisted to `localStorage` under `endpoint-storage` (only `selectedEndpoint`, `authToken`, `user` persist). Holds endpoint, authToken, user, messages, sessions, streaming flags. `login()`/`logout()` reset messages+sessions on account switch.
- **UI primitives** — shadcn/ui (new-york) in `src/components/ui/`. **Custom dark Tailwind palette** (`primary` #FAFAFA, `background` #111113, `background.secondary` #27272A, `accent` #27272A, `muted` #A1A1AA, `brand` #FF4017, `border` rgba(255,255,255,0.2)). **No shadcn `input`/`ring`/`muted-foreground`/`primary-foreground` tokens are defined** — use the custom tokens above. Styling via Tailwind + Framer Motion. Note `TextArea` (`ui/textarea`) is an autosizing **chat** input clamped to ~96px — don't reuse it for long-form editing (use a native `<textarea>`).

### design (Pencil MCP — the UI source of truth)

The `design/` directory is the **authoritative design source** for the whole narratox UI, and it is modified **exclusively through the Pencil MCP server**.

- **`design/narratox.pen`** — the encrypted Pencil design file. **NEVER `Read` / `Grep` / `cat` / `git diff --text` a `.pen` file** — it is encrypted binary; always go through the Pencil MCP tools. The editor state is read with `get_editor_state`, nodes with `batch_get`, structure with `snapshot_layout`, a visual check with `get_screenshot`, tokens with `get_variables`, and all create/update/delete/move operations go through **`batch_design`** (DSL input). Exports via `export_nodes` / `export_html`.
- **Contents** — 38 top-level screen frames at 1440×900 covering every page/state (auth login/register, library main/empty/card-menu/publish, knowledge, dissect main/upload-confirm/log-drawer/result-browser, settings main + dialogs, workspace concept/active/streaming/skeleton + all resource-panel views) + 2 reusable components (`AppSidebar`, `IconRail`). Naming convention `01 Auth Login` / `02 …`. **The old `.pen` was lost and rebuilt from scratch** (see [2026-07-02-ui-redesign-design.md](docs/superpowers/specs/2026-07-02-ui-redesign-design.md)).
- **`design/_exports/`** — HTML/PNG reference exports of individual frames (e.g. `chat-redesign-abcd.html`, `dissect-sidebar-compare.html`, `knowledge-preview.html`). These ARE plain files and may be read normally — but they are **derived snapshots**, not the source; regenerate via `export_nodes`/`export_html` when the `.pen` changes.
- **Design tokens authority** — [docs/superpowers/specs/2026-07-02-ui-redesign-design.md](docs/superpowers/specs/2026-07-02-ui-redesign-design.md) (dark theme, glass morphism, Indigo→Violet `#6366f1→#8b5cf6` gradient, Inter font, full token tables). **Code CSS/Tailwind may lag the design — the spec tokens win.** Pencil MCP v1.1.68 DSL notes: operations use full names (`Insert`/`Update`/`Delete`/`Move`), the param is `input`, icons are `library:"lucide"`, text nodes take no padding/cornerRadius, all text uses Inter.
- **Workflow** — when asked to change the UI design: open `design/narratox.pen` via Pencil MCP → `get_editor_state` → `batch_get`/`snapshot_layout` to find the frame → `batch_design` to mutate → `get_screenshot` to verify → `export_nodes`/`export_html` to refresh `design/_exports/`. Do not hand-edit the exported HTML as a substitute for editing the `.pen`.

### server (NestJS)
Standard Nest modular layout: each feature under `src/<feature>/` as `*.module.ts` + `*.controller.ts` + `*.service.ts` + `dto/`. [src/app.module.ts](server/src/app.module.ts) imports `LoggerModule` (nestjs-pino, first so it captures other modules' init logs), `LoggingModule` (global `AgentLoggerService`), `PrismaModule` (global), `AuthModule`, `AgentosModule`, `NovelModule`, `SettingsModule`, `KnowledgeModule`, `BenchmarkModule`.

- **Auth** ([src/auth/](server/src/auth/)) — JWT. A **global `JwtAuthGuard`** (`APP_GUARD`) protects everything; `@Public()` opts out (register/login/health). `@CurrentUser()` injects `{id, email}`. `ValidationPipe({ whitelist, forbidNonWhitelisted })` is global in [main.ts](server/src/main.ts). All data access is scoped by `user.id` (multi-tenant isolation).
- **Agentos** ([src/agentos/](server/src/agentos/)) — the agent layer. `POST /agents/:id/runs` ([agentos.controller.ts](server/src/agentos/agentos.controller.ts)) feeds a single `DeepAgentService`; the route's `:id` is ignored (the agent is determined by the session's bound novel). The controller walks the langgraph message stream, emits flat `ActivityEvent` frames (`think` / `content` / `tool` / `ActResult`), and closes with a `RunCompleted` frame carrying `aggregateActivities(...)` output. (The old `RunContent` cumulative frames + `WritingChapter` side-channel were removed with the swarm — do not look for them.)
  - **`DeepAgentService`** ([deep-agent.service.ts](server/src/agentos/deep-agent.service.ts)) — builds the agent graph from a **declarative config tree** ([agent-tree.config.ts](server/src/agentos/agent-tree.config.ts)): one main agent + five `task`-delegated orchestrators (`chapter` / `curator` / `worldbuilder` / `outliner` / `character`); `chapter`/`worldbuilder`/`outliner`/`character` each own a writer→critic leaf pair (`writer`/`settler`/`validator` under `chapter`; `wb-writer`/`wb-critic`; `outline-writer`/`outline-critic`; `char-writer`/`char-critic`). `buildAgentGraph` walks `AGENT_TREE` recursively (`buildNode`), resolving prompt (`PROMPTS`), model (`resolveModel`) and tools (`TOOL_REGISTRY`). Uses raw `createAgent` + hand-picked deepagents middleware (**not** `createDeepAgent`, to avoid the filesystem tools it injects). Rebuilt per request (tools closure-inject `userId`/`novelId`). **Adding an agent = adding an `AGENT_TREE` entry, not editing this service.**
  - **Model is per-user + per-agent configurable** — there is no global "active model"; each agent resolves its own. `getModel` reads the user's models via `ModelConfigService` and, per agent, applies an optional `AgentModelOverride` (`modelId` + `temperature`, both nullable = inherit) via `pickAgentConfig`/`resolveModelConfig` (temperature priority: per-agent override → code role default → model default). `buildChatModel` ([model-factory.ts](server/src/agentos/model-factory.ts)) routes by the **Vendor's** `provider`: `openai-compatible` → `ChatOpenAI` (GLM / DeepSeek / Moonshot / Qwen / OpenAI …), `anthropic` → `ChatAnthropic`, `gemini` → `ChatGoogleGenerativeAI`, `deepseek` → `ChatDeepSeek`. Cached by `${modelId}:${updatedAt}:${maxTokens}:${temperature}`; per-role `modelTier` via `MAX_TOKENS_BY_TIER` (long=16k, short=6k). **GLM is no longer hardcoded** — add Vendor+Model in `/settings` (no model configured → the turn throws "尚未配置模型"). The `/settings` agent-model UI is driven by `buildAgentGroups()` ([agent-tree.config.ts](server/src/agentos/agent-tree.config.ts)) — it walks BOTH `AGENT_TREE` and `DISSECT_TREE` so a newly-added agent auto-appears in the config UI with zero extra wiring.
  - **Middleware** — the stack is `createSubAgentMiddleware` (the `task` delegation + nested subagent tree) + `createSummarizationMiddleware` (long-context auto-compression) + `createPatchToolCallsMiddleware` (repair broken tool calls). **No filesystem middleware** — using `createAgent` (not `createDeepAgent`) means `write_file`/`read_file`/`execute` etc. are structurally absent, so no `excludeFilesystemTools` filter is needed.
  - **Agent tools** ([src/agentos/tools/](server/src/agentos/tools/)) — declared per-agent as tool **keys** in `AGENT_TREE` (writing) or `DISSECT_TREE` (dissection), resolved through `TOOL_REGISTRY` ([agent-registry.ts](server/src/agentos/agent-registry.ts)) which maps each name → factory and centralizes service wiring. All LangChain `tool()` factories wrap the resource services directly. **`userId`/`novelId` (writing) and `bookId` (dissection) are closure-injected at build time — never read from LLM input** (security: the model cannot address another user's novel/book). Tool families: chapter write (`append_section`/`replace_text`/`insert_text`/`delete_text`/`clear_chapter`/`set_chapter_title`/`snapshot_chapter`/`restore_chapter`), readers (`get_chapter`/`get_raw_chapter`/`get_reading_chapter`/`list_chapters`/`get_chapter_plan`/`get_outline`/`get_worldview`/`get_world_entry`/`get_character`/`get_characters`/`get_character_history`/`get_arcs`/`get_events`/`get_novel_info`), setters (`set_chapter_plan`/`set_volume`/`set_arc`/`set_master_outline`/`set_world_entry`/`set_character`/`set_references`/`update_novel`), settlement (`write_summary`), recall (`query_memory`), reviews (`report_review`/`report_outline_review`/`report_worldview_review`/`report_character_review`/`report_dissect_review`), knowledge/benchmark (`list_knowledge`/`get_knowledge`/`get_benchmark`/`write_benchmark`/`get_dissect_entries`), and the deterministic guard `check_prose`. **System prompts live in [prompts/*.md](server/src/agentos/prompts/)** — TWO sets: the writing agents (`main`/`chapter-orchestrator`/`writer`/`settler`/`validator`/`curator`/`worldbuilder-*`/`outliner-*`/`character-*`/`main-role-reminder`) loaded by [agent-prompts.ts](server/src/agentos/agent-prompts.ts) into `PROMPTS`, and the dissection agents (`dissect-main`/`chapter-extractor`/`plot-analyst`/`character-extractor`/`style-analyst`/`dissect-critic`) loaded by [dissect-prompts.ts](server/src/agentos/dissect-prompts.ts) into `DISSECT_PROMPTS`. Each `.md` has YAML frontmatter `name`/`key`/`title`/`description` + body = what the LLM receives; the loader (candidate-path fallback: `__dirname/prompts` → `__dirname/../../src/agentos/prompts`) strips frontmatter at module load. **Edit the `.md`, then restart dev** (prompts load into memory at boot — not hot-reloaded). `agent-prompts.spec.ts` locks one feature substring per prompt (migration byte-fidelity guard).
  - **`DissectAgentService`** ([dissect-agent.service.ts](server/src/agentos/dissect-agent.service.ts)) — a SECOND, independent agent runtime for 对标拆解 (book dissection), not part of `DeepAgentService`. Builds its graph from a separate declarative tree **`DISSECT_TREE`** ([dissect-tree.config.ts](server/src/agentos/dissect-tree.config.ts)): a `dissect-main` orchestrator + 5 `task`-delegated subagents (`chapter-extractor` → `plot-analyst` / `character-extractor` / `style-analyst` → `dissect-critic`). Runs **asynchronously in the background** keyed by `bookId` (`startDissect` is fire-and-forget; an in-memory `jobs` map of `{ emitter, abortController }`); the HTTP controller streams the emitter's activity frames to the client as newline-JSON (client disconnect only stops streaming, not the agent). `DissectContextAssembler.forBook(bookId)` provides the dissection system prompt. On boot it marks any stale `RUNNING` jobs (crash leftovers) as `INTERRUPTED`. Model resolution mirrors `DeepAgentService` (per-agent override → active config fallback, same `getModel` cache shape). **Adding a dissection dimension = adding a `DISSECT_TREE` entry + a prompt + a `write_benchmark` type.**
  - `ContextAssembler` ([context-assembler.service.ts](server/src/agentos/context-assembler.service.ts)) — `forSession(userId, sessionId)` returns `{ prompt, novelId }`; the prompt is state-aware (`CONCEPT` → onboarding; `ACTIVE` → writing) and injects 【世界观】(core entries) + 【角色】(tiered: active characters' full profile + dormant roster, via `CharacterService.listForContext` — the fix for "long novels lose character settings") + 【前情】(recent chapter summaries) + 【未回收伏笔】(open `StoryEvent`s) + 【写作参考】 slices.
  - `SessionsService` (`sessions.service.ts`) — chat-thread + transcript storage (per-user scoped). `Session.id` == the LangGraph `thread_id`.
  - `createActivityEmitter` ([activity-emitter.ts](server/src/agentos/activity-emitter.ts)) + `aggregateActivities` translate langgraph message-stream chunks into the flat activity frames.
  - `checkpointer.provider.ts` — `PostgresSaver` in the **`agent_memory` schema** (Prisma manages only `public` — this keeps `prisma migrate dev` drift-free; do not move checkpoint tables to `public`).
  - **Deps:** `@langchain/langgraph`, `@langchain/openai`, `@langchain/anthropic`, `@langchain/google-genai`, `@langchain/deepseek` are direct dependencies. `deepagents` is the agent runtime. (The swarm design was replaced by `createAgent` + `createSubAgentMiddleware`; `@langchain/langgraph-swarm` has been **removed** — do not reintroduce `createSwarm`.)
- **Novel** ([src/novel/](server/src/novel/)) — novel/chapter CRUD + the write path:
  - `NovelController` (`/novels` CRUD, `/novels/:id/chapters[/:cid]`), `NovelService` (creates a Novel + its 1:1 chat `Session` + a seed chapter in one transaction), `ChapterService`.
- **Memory** ([src/memory/](server/src/memory/)) — the long-novel memory services, extracted into their own module (consumed by `NovelModule` + agent tools): `SummaryService` (chapter summaries / 前情), `StoryEventService` (伏笔 hooks lifecycle), `EventService` (discrete plot-event ledger). All scoped by `user.id` + `novelId`/`chapterOrder`.
- **Knowledge** ([src/knowledge/](server/src/knowledge/)) — the **global** writing knowledge base. `KnowledgeService` reads the markdown corpus from `KB_DIR` (env override; default `<repo>/知识库`) at request time — 6 category dirs (人设档案 / 公式模板 / 创作须知 / 拆文案例 / 词汇素材库 / 方法论教程), each `.md` parsed for frontmatter `tags` + a `> **一句话**：…` description. **JWT-protected but NOT user-scoped** (shared corpus). Routes: `GET /knowledge` (filter by category/tag/search), `GET /knowledge/:id`. The `list_knowledge` / `get_knowledge` agent tools read this; the curated 拆书教程 corpus under `知识库/整理版/` is the indexed subset.
- **Benchmark** ([src/benchmark/](server/src/benchmark/)) — the 对标拆解 (dissection) persistence + HTTP entry. `BenchmarkService` (upload → `splitChapters` 纯函数切章 → `BenchmarkBook`/`BenchmarkEntry` CRUD; `writeEntry` appends a typed entry; `markInterruptedOnBoot`), `BenchmarkController` (`/benchmarks`: `POST /upload` multipart → 切章落库; `GET /` list / `GET /:id` detail; `POST /:id/dissect` 启动后台拆解 + 同连接流化活动帧; `GET /:id/stream` 断线重连; `DELETE /:id`). Dissection is an async background job — the HTTP connection only streams the emitter's activity frames; client disconnect stops streaming but not the agent. `BenchmarkEntry.type` is one of `CHAPTER`/`PLOT`/`RHYTHM`/`EMOTION`/`CHARACTER`/`STYLE` (the 6 dissection dimensions).
- **Logging** ([src/logging/](server/src/logging/)) — structured logging via `nestjs-pino` (`LoggerModule.forRoot` is first in `AppModule` so it captures other modules' init logs) + a global `AgentLoggerService` (`LoggingModule` is `@Global`). Used for agent activity/tracing.
- **Settings** ([src/settings/](server/src/settings/)) — per-user provider/model + per-agent model + voice-profile configs. **Vendor/Model two-tier** (replaced the old flat `ModelConfig`): `Vendor` (provider/baseUrl/apiKey — credentials shared across models of the same provider) → `Model` (model id/name/temperature, belongs to a Vendor). `ModelConfigService` serves a masked shape to the API (`hasApiKey`, never the raw key) and the full row to the agent services. Routes: `/settings/vendors` (+ `/vendors/:vid/models`), `/settings/models/:id/activate` (sets `User.activeModelId`), **`/settings/agent-models`** (+ `/agent-models/:agentKey`) — the `AgentModelOverride` per-agent modelId+temperature config (keys auto-derived from `AGENT_TREE`+`DISSECT_TREE` via `buildAgentGroups`), and **`/settings/voice-profiles`** — the `VoiceProfile` per-user 作者画像 library (CRUD + `generate`). Each `Novel` picks one `VoiceProfile` (`Novel.voiceProfileId`) — writer/validator inject that novel's selected profile. API keys are stored plaintext — a known limitation; encryption-at-rest is a deferred hardening.
- **Data model** (Prisma, [prisma/schema.prisma](server/prisma/schema.prisma)) — `User` (has `activeModelId` → `Model`), `Vendor` → `Model` (provider credentials + model rows; replaced the old flat `ModelConfig`), `AgentModelOverride` (per-agent `modelId`+`temperature`), `VoiceProfile` (per-user 作者画像 library; `Novel.voiceProfileId` picks one), `Session` (chat thread), `Message` (chat history), `Novel` (1:1 `Session`, has `settings` JSON, **`status: CONCEPT|ACTIVE`**), `Chapter` (manuscript, `status: DRAFT|COMMITTED`), `ChapterSummary` (前情), `StoryEvent` (伏笔 hooks), `Event` (plot-event ledger, `significance MAJOR|MINOR`), `CharacterChange` (`significance MAJOR|MINOR`), `Volume` (synopsis + rolling `arcSummary` + `bridge`/`mainProgress`), `MasterOutline` (1:1 Novel, book-level theme/mainLine/ending/powerProgression/hiddenLines/threeAct), `Arc` (sub-volume segment), `ChapterOutline` (细纲, `status DRAFT|APPROVED`), `WorldEntry`, `NovelReference` (`injectTo` = role name), `Character` (+ `growth`/`flaw` stable-profile fields), and the dissection pair `BenchmarkBook` / `BenchmarkEntry` (`status PENDING|RUNNING|DONE|FAILED|INTERRUPTED`; `type CHAPTER|PLOT|RHYTHM|EMOTION|CHARACTER|STYLE`). **`Novel.status` lifecycle**: a bare novel created via 「新建小说」starts `CONCEPT` (info incomplete, no chapter content); the first chapter content flips it to `ACTIVE`. Historical rows default to `ACTIVE`. **Chat (`Message`) ≠ manuscript (`Chapter.content`).** The writer subagent edits `Chapter.content` directly via tools (no manual copy). **Prisma 7 `generate` gotcha**: after editing `schema.prisma`, `migrate dev` does NOT auto-regenerate the client — run `pnpm --dir server prisma generate` manually (see [prisma.config.ts](server/prisma.config.ts); CLI takes no `--schema` flag).
- **`server/tsconfig.json`** uses `"types": ["node", "jest"]` — `node` must stay so `process.env` type-checks under `nest build`. **`tsconfig.build.json` pins `"rootDir": "./src"`** (and excludes `scripts/`/`test/`/specs) — without it tsc inferred rootDir as the project root and nested output under `dist/src/`, which broke `start:prod` (`node dist/main`) and stranded `prompts/*.md` away from the compiled JS. A `postbuild` step (`cp -R src/agentos/prompts dist/agentos/`) copies the markdown into `dist/` for production (the loader's src-fallback covers jest/dev; this covers dist-only deploys). `nest-cli` `assets` was tried but the v11 asset copier did not fire in this project — the `postbuild` copy is used instead.
